This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

**Run the API server:**
```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

**Run Celery worker** (required for async document processing):
```bash
celery -A app.celery_app worker -Q document_queue -l info
```

**Train ML classifier:**
```bash
python app/ml/train_classifier.py
```

**Database migrations** (Alembic is installed):
```bash
alembic upgrade head
```

**Run all tests:**
```bash
pytest tests/
```

**Run a single test file or test:**
```bash
pytest tests/test_documents.py
pytest tests/test_documents.py::test_upload_document
```

Tests require a PostgreSQL database named `docintel_test` (same host/credentials as dev). Override with `TEST_DATABASE_URL` env var. Celery and file storage are mocked automatically — no broker or filesystem needed for tests.

## Architecture

This is a **FastAPI + Celery** document intelligence platform with multi-tenant organization support.

### Request Flow

1. **HTTP Request** → FastAPI routes (`app/routes/`) validate JWT via `app/auth/dependencies.py`
2. **Document upload** → file saved to `storage/documents/`, Celery task queued
3. **Celery worker** → `app/tasks.py` → `app/services/ai_pipeline.py` orchestrates full processing
4. **Results** stored as JSONB in `extracted_data` table with confidence scores

### AI Processing Pipeline (`app/services/`)

Documents go through this sequence in `ai_pipeline.py`:
1. `ocr_service.py` — extract text (pdfplumber for PDFs, Tesseract for images)
2. `nlp_service.py` — named entity recognition via spaCy (`en_core_web_sm`)
3. `ml_classifier_service.py` — classify document type (LogisticRegression + TF-IDF from `app/ml/*.joblib`)
4. `document_classifier.py` — rule-based pattern fallback
5. `pipeline_router.py` — routes to `pipelines/invoice_pipeline.py` or `pipelines/generic_pipeline.py`
6. `confidence_service.py` — hybrid score: `0.7 * ML_prob + 0.3 * rule_score`
7. Results stored in `extracted_data` table; `documents.searchable_text` is populated for full-text search

### Search

Two search modes in `search_service.py`:
- **Keyword search** — PostgreSQL `to_tsvector` / `plainto_tsquery` on `documents.searchable_text`
- **Semantic search** — pgvector cosine similarity on `extracted_data.embedding`; embeddings generated by `embedding_service.py` using `sentence-transformers/all-MiniLM-L6-v2` (lazy-loaded, ~90 MB)

### Multi-Tenancy & RBAC

- `OrganizationUser` bridge table holds `role` for each user-org pair
- Three roles (`OrgRole` enum in `app/models/organization_user.py`): `admin`, `analyst`, `viewer`
- Routes use `require_role(*roles)` from `app/auth/rbac.py` — a dependency factory that returns the membership record on success
- Upload and reprocess: admin + analyst. Delete: admin only. Read/list/search/export: all roles.

### Versioning

Each reprocessing run appends a new `ExtractedData` row with an incremented `version_number`. `GET /documents/results/{id}` returns the highest version; `GET /documents/results/{id}/versions` returns all versions ascending.

### Infrastructure Dependencies

| Service | Default |
|---------|---------|
| PostgreSQL | `localhost:5433`, db `docintel` |
| Redis (Celery broker) | `localhost:6379/0` |
| Tesseract binary | `C:\Program Files\Tesseract-OCR\tesseract.exe` |

Database tables are auto-created at startup via `Base.metadata.create_all()` in `app/main.py`, plus an idempotent `ALTER TABLE` to add `version_number` for pre-migration databases.

### Key Config

- `app/config.py` — reads `.env`, exposes `Settings` singleton
- `app/database.py` — SQLAlchemy engine and `get_db` session dependency
- `app/celery_app.py` — Celery instance; task queue name is `document_queue`

### Document Status Lifecycle

`uploaded` → `queued` → `processing` → `completed` | `failed`

Status is an enum on the `Document` model (`app/models/document.py`).
